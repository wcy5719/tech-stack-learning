## 第三章：核心模块详解

### 3.1 二级缓存（系统最快的部分）

#### 用图书馆来理解

假设你去图书馆借书：

```
第一反应：我书桌上的书（L1缓存，< 1毫秒）
  ↓ 没有
第二反应：图书馆书架（L2缓存，~10毫秒）
  ↓ 没有
第三反应：向其他图书馆借阅（vLLM推理，~20秒）
```

#### 代码流程

```java
public ChatResponse getAnswer(String question) {
    // 1. 先看自己书桌（L1缓存）
    ChatResponse answer = desk.get(question);
    if (answer != null) {
        return answer;  // 找到了！超快！
    }
    
    // 2. 再看图书馆书架（L2缓存）
    answer = library.get(question);
    if (answer != null) {
        desk.put(question, answer);  // 找到了，顺便放书桌上
        return answer;
    }
    
    // 3. 最后只能问AI了（慢...）
    answer = askAI(question);
    desk.put(question, answer);
    library.put(question, answer);
    return answer;
}
```

#### 为什么需要两层？

| 场景     | 只用L1 | 只用L2 | 用L1+L2 |
| ------ | ---- | ---- | ------ |
| 速度     | 最快   | 中等   | 最快     |
| 多服务器共享 | ❌ 不行 | ✅ 可以 | ✅ 可以   |
| 内存占用   | 很大   | 小    | 小      |

**最佳方案**：L1负责快，L2负责共享，两者配合！

### 3.2 布隆过滤器（聪明的门卫）

#### 用快递柜来理解

假设快递柜有1000个格子，你问："有没有我的快递？"

**笨办法**：挨个格子找一遍 → 1000次检查

**布隆过滤器**：一个智能系统，问一下就告诉你"可能有"或"肯定没有"

```
用户问："订单查询的答案在缓存里吗？"

布隆过滤器：
  - "肯定没有" → 不用查缓存了，直接让AI回答
  - "可能有" → 去缓存查一下确认
```

#### 特点

- **速度快**：O(1)复杂度，不管存了多少数据，都是一瞬间
- **误判率**：说"可能有"时，实际可能没有（1%概率）
- **准确率**：说"没有"时，那就真的没有（100%准确）

**就像门卫说"这个人没来过" → 绝对准确；说"来过" → 可能是认错人了**

### 3.3 异步处理（最关键的高并发方案）

#### 用餐厅点餐来理解

**同步模式（慢）：**

```
服务员A：接待顾客1 → 等厨师做好 → 端给顾客1 → 接待顾客2
服务员B：等着...等着...等着...

结果：200个服务员，同时只能服务200个顾客
```

**异步模式（快）：**

```
服务员A：接待顾客1 → 下单 → 立刻接待顾客2 → 立刻接待顾客3
服务员B：接待顾客N → 下单 → 继续接待...

厨师在厨房做菜，做好后叫服务员来端

结果：200个服务员，可以同时服务1000个顾客！
```

#### 代码实现

**Controller层（服务员接单）：**

```java
@PostMapping("/chat")
public DeferredResult<ChatResponse> chat(ChatRequest request) {
    // 创建一个"待处理"的结果对象
    DeferredResult<ChatResponse> result = new DeferredResult<>();
    
    // 提交给厨师（异步线程池）
    CompletableFuture<ChatResponse> future = asyncService.process(request);
    
    // 厨师做好了，设置结果
    future.whenComplete((response, error) -> {
        result.setResult(response);
    });
    
    // 服务员立即返回（不等待！）
    return result;
}
```

**Service层（厨师做菜）：**

```java
@Async("chatTaskExecutor")  // 在独立线程池中执行
public CompletableFuture<ChatResponse> process(ChatRequest request) {
    // AI思考（耗时20秒）
    ChatResponse answer = aiService.answer(request.getMessage());
    return CompletableFuture.completedFuture(answer);
}
```

#### 效果对比

| 模式 | 500人同时提问      | 结果   |
| -- | ------------- | ---- |
| 同步 | 200人等待，300人超时 | ❌ 失败 |
| 异步 | 500人全部成功      | ✅ 成功 |

### 3.4 限流（保护系统的门卫）

#### 用游乐园来理解

游乐园有个热门项目，每次只能进10个人。

**不限流**：

```
1000个人同时冲进去 → 拥挤 → 踩踏事故 → 系统崩溃
```

**限流**：

```
门卫：一分钟只能进100个人，其他人排队
结果：安全有序，系统稳定
```

#### 代码实现

```java
public boolean isAllowed(String ip) {
    // Redis计数器：这个IP一分钟内访问了几次？
    Long count = redis.increment("rate_limit:" + ip);
    
    if (count == 1) {
        // 第一次，设置1分钟后过期
        redis.expire("rate_limit:" + ip, 60, SECONDS);
    }
    
    // 超过100次？对不起，限流！
    return count <= 100;
}
```

### 3.5 熔断器（系统的保险丝）

#### 用电路保险丝来理解

```
正常情况：电流通过 → 电器工作

过载情况：电流过大 → 保险丝熔断 → 保护电路

```

#### 在我们的系统中

```

         "AI服务暂时不可用，请稍后再试"

30秒后：熔断器半开 → 尝试几个请求 → vLLM恢复了？ → 关闭熔断器
```

#### 三种状态

```
关闭(Closed) ────失败太多────→ 打开(Open)
    ↑                              ↓
    │                        等30秒
    │                              ↓
    └───────探测成功─────── 半开(Half-Open)
```

### 3.7 多轮对话（记住上下文）

#### 用记忆好的客服来理解

**没有多轮对话：**

```
用户："我的订单什么时候到？"
AI："您的订单预计明天到达。"

用户："那个呢？"  ← 哪个？AI不知道
AI："请问您指的是什么？"
```

**有多轮对话：**

```
用户："我的订单什么时候到？"
AI："您的订单预计明天到达。"
（系统记住：用户问了订单物流）

用户："那个呢？"
AI："您指的是订单物流吗？预计明天到达。"
（系统知道"那个"指的是刚才的订单）
```

#### 代码实现

```java
public void addMessage(String sessionId, String role, String content) {
    // 获取对话历史
    List<Message> history = getHistory(sessionId);
    
    // 添加新消息
    history.add(new Message(role, content));
    
    // 只保留最近10条（防止内存爆炸）
    if (history.size() > 10) {
        history = history.subList(history.size() - 10, history.size());
    }
    
    // 存到Redis，30分钟过期
    redis.set("conversation:" + sessionId, history, 30, MINUTES);
}
```

## 第七章：常见问题解答

### Q1：为什么要用缓存？直接让AI回答不行吗？

**A：** 可以，但是...

```
不用缓存：
  - 每次都要等20秒
  - GPU资源浪费
  - 用户等得不耐烦
  - 成本高

使用缓存：
  - 相同问题3毫秒返回
  - 节省GPU资源
  - 用户体验好
  - 成本低
```

**就像你问助手"1+1等于几"，助手不会每次都重新算，而是直接回答"2"**

### Q2：异步处理是什么？为什么要异步？

**A：**

```
同步 = 你去餐厅点餐，站在柜台等厨师做好才走

异步 = 你去餐厅点餐，拿到号码牌去逛街，做好了叫你
```

AI推理需要20秒，如果同步等，200个用户同时提问，系统就卡死了。
异步可以让系统持续接收新请求，不被慢请求阻塞。

### Q3：布隆过滤器为什么快？

**A：**

```
传统方法：在1000个数据中查找 → 最多查1000次

布隆过滤器：一个哈希函数 → 查1次就知道"肯定没有"或"可能有"
```

**就像你有一个智能通讯录，问"张三在吗？"，系统瞬间回答"不在"或"可能在"**

### Q4：熔断器有什么用？

**A：**

```
没有熔断器：
  AI服务挂了 → 所有请求都等20秒超时 → 系统卡死 → 雪崩效应

有熔断器：
  AI服务挂了 → 检测到失败率高 → 熔断器打开 → 立即返回降级回复 → 保护系统
```

**就像家里的保险丝，电流过大时自动断开，保护整个电路**

### Q5：500人同时提问，系统为什么没崩溃？

**A：** 四个防护层起作用：

1. **限流**：恶意刷接口的被拦截
2. **缓存**：70%的问题直接返回，不用问AI
3. **异步**：Tomcat线程不阻塞，持续接收请求
4. **熔断器**：AI服务挂了也不影响缓存命中的请求

**就像一个有安保、快速通道、VIP服务、备用电源的商场，人再多也能有序运转**
