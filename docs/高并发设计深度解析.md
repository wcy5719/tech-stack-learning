# AI客服系统 - 高并发设计深度解析

---

## 一、高并发场景分析

### 1.1 业务挑战

**AI客服系统的特殊性：**

```
普通Web请求：~50ms 响应
AI推理请求：~20,000ms 响应（400倍差距）

问题：如果采用同步处理，Tomcat线程会被长期占用
```

**典型高并发场景：**
1. **促销活动**：瞬时大量用户咨询
2. **服务故障**：大量用户同时反馈问题
3. **日常高峰**：午间/晚间咨询高峰期
4. **恶意攻击**：爬虫或恶意刷接口

### 1.2 系统瓶颈识别

```
┌─────────────────────────────────────────────────────────┐
│                    传统同步架构                          │
│                                                         │
│  请求 → Tomcat线程(200) → 等待vLLM(20s) → 返回          │
│                                                         │
│  问题1: 200个线程 × 20秒 = 每秒只能处理10个请求          │
│  问题2: 新请求被阻塞，等待队列越来越长                    │
│  问题3: 超时请求堆积，系统逐渐崩溃                        │
└─────────────────────────────────────────────────────────┘
```

## 二、高并发解决方案

### 2.1 总体策略

```
┌──────────────────────────────────────────────────────────┐
│                   高并发四层防护                          │
│                                                          │
│  第一层：接入层限流（IP Rate Limiting）                   │
│  第二层：缓存层拦截（二级缓存 + Bloom Filter）            │
│  第三层：异步化处理（DeferredResult + @Async）            │
│  第四层：熔断器保护（Resilience4j CircuitBreaker）        │
└──────────────────────────────────────────────────────────┘
```

### 2.2 第一层：IP限流

**实现原理：**

```java
@Service
@RequiredArgsConstructor
public class IpRateLimitService {

    private final RedisTemplate<String, Object> redisTemplate;
    private static final String RATE_LIMIT_KEY = "rate_limit:";
    
    public boolean isAllowed(String ip, int limit) {
        String key = RATE_LIMIT_KEY + ip;
        
        // Redis原子计数器
        Long count = redisTemplate.opsForValue().increment(key);
        
        if (count == 1) {
            // 首次请求，设置1分钟过期
            redisTemplate.expire(key, 60, TimeUnit.SECONDS);
        }
        
        // 超过限制则拒绝
        return count <= limit;
    }
}

@Component
@RequiredArgsConstructor
public class RateLimitInterceptor implements HandlerInterceptor {
    
    private final IpRateLimitService ipRateLimitService;
    private final AiMetricsCollector metricsCollector;
    
    @Override
    public boolean preHandle(HttpServletRequest request, HttpServletResponse response, 
                           Object handler) throws Exception {
        String clientIp = getClientIp(request);
        
        if (!ipRateLimitService.isAllowed(clientIp, 100)) {
            // 记录限流指标
            metricsCollector.recordRateLimited();
            
            // 返回429
            response.setStatus(429);
            response.getWriter().write(
                "{\"error\":\"Too Many Requests\",\"message\":\"请求频率过高\"}"
            );
            return false;
        }
        
        return true;
    }
}
```

**关键设计：**
1. **滑动窗口**：60秒时间窗口，自动过期
2. **原子操作**：Redis increment保证并发安全
3. **指标收集**：限流请求计入Prometheus监控

### 2.3 第二层：二级缓存架构

**缓存层级设计：**

```
请求到达
    ↓
Bloom Filter（O(1)判断key是否存在）
    ↓ 不存在 → 直接跳过缓存检查
    ↓ 存在
L1 Caffeine缓存（进程内，< 1ms）
    ↓ 命中 → 返回（结束）
    ↓ 未命中
L2 Redis缓存（分布式，~10ms）
    ↓ 命中 → 返回 + 回填L1（结束）
    ↓ 未命中
调用vLLM推理（20s）
    ↓
写入L1和L2缓存
```

**核心实现：**

```java
@Service
public class TwoLevelCacheService {

    // L1: Caffeine本地缓存
    private final Cache<String, ChatResponse> localCache;
    
    // L2: Redis分布式缓存
    private final RedisTemplate<String, Object> redisTemplate;
    
    private static final String REDIS_KEY_PREFIX = "ai_response:";
    private static final long LOCAL_CACHE_TTL = 60;      // 1分钟
    private static final long REDIS_CACHE_TTL = 3600;    // 1小时
    
    public ChatResponse get(String key) {
        // 1. L1缓存查询
        ChatResponse response = localCache.getIfPresent(key);
        if (response != null) {
            response.setCached(true);
            return response;
        }
        
        // 2. L2缓存查询
        response = getFromRedis(key);
        if (response != null) {
            // L2命中，回填L1
            localCache.put(key, response);
            response.setCached(true);
            return response;
        }
        
        return null;
    }
    
    public void put(String key, ChatResponse value) {
        // 写入L1
        localCache.put(key, value);
        
        // 写入L2（JSON序列化）
        String json = objectMapper.writeValueAsString(value);
        redisTemplate.opsForValue().set(
            REDIS_KEY_PREFIX + key,
            json,
            REDIS_CACHE_TTL,
            TimeUnit.SECONDS
        );
    }
}
```

**Bloom Filter优化：**

```java
@Service
public class BloomFilterService {

    private BloomFilter<String> bloomFilter;
    
    @PostConstruct
    public void init() {
        // 预期10000个元素，误判率1%
        bloomFilter = BloomFilter.create(
            Funnels.stringFunnel(Charset.defaultCharset()),
            10000,
            0.01
        );
    }
    
    public boolean mightContain(String key) {
        return bloomFilter.mightContain(key);
    }
    
    public void put(String key) {
        bloomFilter.put(key);
    }
}
```

**为什么用Bloom Filter？**

```
不用Bloom Filter：
  每个请求都要查Redis（10ms）× 1000并发 = 大量无效查询

使用Bloom Filter：
  O(1)内存判断 → key不存在直接跳过 → 节省90%+无效查询
```

### 2.4 第三层：异步处理（核心）

#### 2.4.1 同步 vs 异步对比

**同步模式（问题）：**

```
时间轴（200个Tomcat线程）：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
线程1: [请求1----20s----→返回]
线程2: [请求2----20s----→返回]
...
线程200: [请求200--20s--→返回]

请求201: [等待...等待...等待...超时❌]

结果：每秒最多处理 200/20 = 10个请求
```

**异步模式（解决方案）：**

```
时间轴：
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
Tomcat线程1: [接收请求1→立即释放]          [接收请求2→立即释放]
Tomcat线程2: [接收请求3→立即释放]          [接收请求4→立即释放]
...
Tomcat线程200: [接收请求N→立即释放]

ai-chat-async-1:          [----vLLM-20s----→回调设置结果]
ai-chat-async-2:          [----vLLM-20s----→回调设置结果]
...

结果：Tomcat可以持续接收请求，无阻塞
```

#### 2.4.2 DeferredResult 实现

**Controller层：**

```java
@PostMapping("/chat")
public DeferredResult<ResponseEntity<ChatResponse>> chat(@Valid @RequestBody ChatRequest request) {
    // 创建DeferredResult，设置10分钟超时
    DeferredResult<ResponseEntity<ChatResponse>> deferredResult = 
        new DeferredResult<>(600_000L);
    
    // 1. 超时处理
    deferredResult.onTimeout(() -> {
        ChatResponse timeoutResponse = ChatResponse.builder()
                .response("请求处理超时，请稍后再试")
                .build();
        deferredResult.setResult(ResponseEntity.ok(timeoutResponse));
    });
    
    // 2. 超时回调
    deferredResult.onTimeout(() -> {
        log.warn("请求处理超时: userId={}", request.getUserId());
    });
    
    // 3. 完成回调
    deferredResult.onCompletion(() -> {
        log.debug("请求处理完成: userId={}", request.getUserId());
    });
    
    // 4. 启动异步处理
    CompletableFuture<ChatResponse> future = asyncChatService.processChatAsync(request);
    
    // 5. 异步完成后的回调
    future.whenComplete((response, error) -> {
        if (error != null) {
            // 错误处理
            ChatResponse errorResponse = ChatResponse.builder()
                    .response("处理失败: " + error.getMessage())
                    .build();
            deferredResult.setResult(ResponseEntity.status(500).body(errorResponse));
        } else {
            // 成功处理
            deferredResult.setResult(ResponseEntity.ok(response));
        }
    });
    
    // 6. 立即返回DeferredResult（不阻塞！）
    return deferredResult;
}
```

**关键理解：**

```java
// 这个方法做了什么？
// 1. 创建DeferredResult对象
// 2. 注册超时和完成回调
// 3. 启动异步任务（提交到线程池）
// 4. 返回DeferredResult（此时请求还在处理中）

// Spring的处理：
// - 收到DeferredResult后，Tomcat线程立即释放
// - HTTP连接保持打开（Servlet 3.0异步支持）
// - 等待异步任务调用 setResult() 后才返回响应
```

#### 2.4.3 @Async 异步执行

**Service层：**

```java
@Service
@RequiredArgsConstructor
public class AsyncChatService {

    private final AiCustomerService aiCustomerService;
    
    /**
     * 异步处理聊天请求
     * @Async 注解使方法在独立线程池中执行
     */
    @Async("chatTaskExecutor")
    public CompletableFuture<ChatResponse> processChatAsync(ChatRequest request) {
        log.info("异步处理开始: userId={}", request.getUserId());
        long startTime = System.currentTimeMillis();
        
        try {
            // 实际处理（可能包含vLLM调用，耗时20s）
            ChatResponse response = aiCustomerService.processChat(request);
            
            long duration = System.currentTimeMillis() - startTime;
            log.info("异步处理完成: userId={}, 耗时={}ms", request.getUserId(), duration);
            
            return CompletableFuture.completedFuture(response);
        } catch (Exception e) {
            log.error("异步处理失败: userId={}, error={}", request.getUserId(), e.getMessage());
            return CompletableFuture.failedFuture(e);
        }
    }
}
```

#### 2.4.4 线程池配置

```java
@Configuration
@EnableAsync
public class AsyncConfig {

    @Bean(name = "chatTaskExecutor")
    public Executor chatTaskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        
        // 核心参数设计
        executor.setCorePoolSize(
            Runtime.getRuntime().availableProcessors() * 2
        );  // 核心线程数 = CPU核数 × 2
        
        executor.setMaxPoolSize(
            Runtime.getRuntime().availableProcessors() * 8
        );  // 最大线程数 = CPU核数 × 8
        
        executor.setQueueCapacity(500);  // 队列容量500
        
        executor.setThreadNamePrefix("ai-chat-async-");  // 线程名前缀
        
        // 拒绝策略：CallerRunsPolicy
        // 当队列满时，由调用者线程执行（背压机制）
        executor.setRejectedExecutionHandler(
            new ThreadPoolExecutor.CallerRunsPolicy()
        );
        
        // 优雅关闭
        executor.setWaitForTasksToCompleteOnShutdown(true);
        executor.setAwaitTerminationSeconds(60);
        
        executor.initialize();
        return executor;
    }
}
```

**线程池参数详解：**

| 参数 | 值（假设8核CPU） | 说明 |
|------|-----------------|------|
| CorePoolSize | 16 | 常驻线程数，处理日常请求 |
| MaxPoolSize | 64 | 高峰期最大线程数 |
| QueueCapacity | 500 | 缓冲队列，削峰填谷 |
| 拒绝策略 | CallerRunsPolicy | 队列满时降速，不丢请求 |

**线程池工作流程：**

```
新任务到达
    ↓
核心线程数未满？ → 是 → 创建新线程处理
    ↓ 否
队列未满？ → 是 → 放入队列等待
    ↓ 否
最大线程数未满？ → 是 → 创建新线程处理
    ↓ 否
触发拒绝策略 → CallerRunsPolicy → 调用者线程执行
```

### 2.5 第四层：熔断器保护

#### 2.5.1 CircuitBreaker 工作原理

```
状态机转换：

    关闭(Closed) ─────失败率>60%─────→ 打开(Open)
        ↑                                   ↓
        │                            等待30秒
        │                                   ↓
        └─────探测成功───────── 半开(Half-Open)
                                      ↓
                                 探测失败 → 回到Open
```

**配置详解：**

```yaml
resilience4j:
  circuitbreaker:
    instances:
      aiService:
        slidingWindowSize: 20           # 统计最近20次调用
        minimumNumberOfCalls: 10        # 至少调用10次才开始统计
        failureRateThreshold: 60        # 失败率超过60%触发熔断
        slowCallRateThreshold: 80       # 慢调用率超过80%触发熔断
        slowCallDurationThreshold: 45s  # 超过45秒算慢调用
        waitDurationInOpenState: 30s    # 熔断后等待30秒再尝试
        permittedNumberOfCallsInHalfOpenState: 5  # 半开状态允许5次探测
```

#### 2.5.2 代码实现

```java
@CircuitBreaker(name = "aiService", fallbackMethod = "callAiServiceFallback")
@Retry(name = "aiService")
private String callAiService(String fullMessage, ChatRequest request) {
    // 记录指标
    Timer.Sample inferenceTimer = metricsCollector.startInferenceTimer();
    
    try {
        // vLLM调用（可能失败或超时）
        String result = vllmService.chatComplete(buildMessages(fullMessage, request));
        
        // 记录成功指标
        metricsCollector.recordInferenceTime(inferenceTimer);
        return result;
    } catch (Exception e) {
        // 记录失败指标
        metricsCollector.recordError("inference_failed");
        throw e;  // 抛出异常，由CircuitBreaker捕获
    }
}

private String callAiServiceFallback(String fullMessage, ChatRequest request, Throwable throwable) {
    log.warn("vLLM调用失败，使用降级回复: {}", throwable.getMessage());
    metricsCollector.recordError("vllm_call_fallback");
    
    // 降级响应
    return "当前AI推理服务暂时不可用，已切换到基础回复模式。";
}
```

## 五、关键设计模式

### 5.1 DeferredResult 模式

```java
// 传统同步Controller
@PostMapping("/sync")
public ResponseEntity<ChatResponse> sync(@RequestBody ChatRequest request) {
    ChatResponse response = service.process(request);  // 阻塞20s
    return ResponseEntity.ok(response);
}

// 异步DeferredResult模式
@PostMapping("/async")
public DeferredResult<ResponseEntity<ChatResponse>> async(@RequestBody ChatRequest request) {
    DeferredResult<ResponseEntity<ChatResponse>> result = new DeferredResult<>();
    
    executor.submit(() -> {
        ChatResponse response = service.process(request);  // 不阻塞Tomcat
        result.setResult(ResponseEntity.ok(response));
    });
    
    return result;  // 立即返回
}
```

### 5.2 CompletableFuture 链式调用

```java
CompletableFuture<ChatResponse> future = asyncChatService.processChatAsync(request);

future.whenComplete((response, error) -> {
    // 异步完成后的处理
});

future.exceptionally(error -> {
    // 异常处理
    return fallbackResponse;
});

future.thenApply(response -> {
    // 结果转换
    return transform(response);
});
```

### 5.3 背压机制

```java
// CallerRunsPolicy 拒绝策略
executor.setRejectedExecutionHandler(
    new ThreadPoolExecutor.CallerRunsPolicy()
);

// 效果：
// 当队列满时，新任务由调用者线程执行
// 相当于"减速"，不会丢失请求
// Tomcat线程会执行任务 → 处理变慢 → 客户端等待时间增长 → 自然限流
```

## 七、面试高频问题

### Q1: 为什么选择DeferredResult而不是@Async？

**A:** 
- `@Async`适用于不需要同步返回结果的场景（如发送邮件）
- `DeferredResult`适用于需要异步处理但最终要返回结果的场景
- 本项目需要返回AI响应，所以用`DeferredResult`
- 实际实现中，两者结合使用：Controller用DeferredResult，Service用@Async

### Q2: ThreadPoolExecutor参数如何设计？

**A:**
```
核心线程数 = CPU核数 × 2  （AI调用是IO密集型，需要更多线程）
最大线程数 = CPU核数 × 8  （留有余量应对突发）
队列容量 = 500             （缓冲峰值请求）
拒绝策略 = CallerRunsPolicy（背压，不丢请求）
```

### Q3: 如何保证异步任务的异常不会丢失？

**A:**
```java
// 错误做法
@Async
public void process() {
    // 异常会被吞掉
    throw new RuntimeException();
}

// 正确做法
@Async
public CompletableFuture<Void> process() {
    try {
        // 处理逻辑
    } catch (Exception e) {
        return CompletableFuture.failedFuture(e);
    }
    return CompletableFuture.completedFuture(null);
}

// Controller层捕获
future.whenComplete((result, error) -> {
    if (error != null) {
        // 处理异常
    }
});
```

### Q4: 500并发测试中为什么没有请求丢失？

**A:**
1. **异步架构**：Tomcat线程不阻塞，持续接收请求
2. **队列缓冲**：500容量的队列吸收峰值
3. **背压机制**：CallerRunsPolicy确保不丢请求
4. **缓存拦截**：69%的请求直接被缓存处理，不进入队列
