## 三、核心功能深度剖析

### 3.1 二级缓存架构（核心亮点）

#### 设计思想
```
请求 → Bloom Filter → L1 Caffeine → L2 Redis → vLLM推理
           ↓              ↓            ↓
         快速拒绝        < 1ms       ~10ms
```

**为什么用二级缓存？**
- **L1 Caffeine**：进程内缓存，纳秒级响应，适合热点数据
- **L2 Redis**：分布式缓存，支持多实例共享，适合全量缓存
- **Bloom Filter**：快速判断key是否存在，避免无效查询

#### 代码实现关键点

```java
public ChatResponse get(String key) {
    // 1. L1: 本地缓存（纳秒级）
    ChatResponse response = localCache.getIfPresent(key);
    if (response != null) {
        response.setCached(true);
        return response;
    }

    // 2. L2: Redis缓存（毫秒级）
    response = getFromRedis(key);
    if (response != null) {
        localCache.put(key, response); // L2回写L1
        response.setCached(true);
        return response;
    }

    // 3. 缓存未命中，返回null
    return null;
}
```

#### 性能数据
| 场景 | 响应时间 | 说明 |
|------|----------|------|
| L1缓存命中 | < 1ms | 进程内直接返回 |
| L2缓存命中 | ~10ms | Redis网络IO |
| 缓存未命中 | 20-300s | vLLM推理 |

#### 面试常见问题
**Q: 如何保证L1和L2缓存一致性？**  
A: 使用Redis Pub/Sub发布缓存更新消息，其他实例收到消息后清除本地L1缓存（CacheSyncService实现）。

**Q: Bloom Filter有什么优势？**  
A: O(1)时间复杂度判断key是否存在，节省大量无效的Redis查询，内存占用极小。

### 3.2 异步处理架构（高并发关键）

#### 同步 vs 异步对比

**同步模式的问题：**
```
请求 → Tomcat线程 → 等待vLLM(20s) → 返回
问题：200个Tomcat线程很快耗尽，新请求被阻塞
```

**异步模式的改进：**
```
请求 → Tomcat立即释放 → DeferredResult等待
     → ai-chat-async线程池 → vLLM推理 → 回调设置结果
优势：Tomcat可处理更多并发请求
```

#### 核心实现

```java
// Controller层：使用DeferredResult
@PostMapping("/chat")
public DeferredResult<ResponseEntity<ChatResponse>> chat(@RequestBody ChatRequest request) {
    DeferredResult<ResponseEntity<ChatResponse>> deferredResult = new DeferredResult<>(600_000L);
    
    // 设置超时处理
    deferredResult.onTimeout(() -> {
        deferredResult.setResult(ResponseEntity.ok(timeoutResponse));
    });
    
    // 异步处理
    CompletableFuture<ChatResponse> future = asyncChatService.processChatAsync(request);
    
    // 完成回调
    future.whenComplete((response, error) -> {
        if (error != null) {
            deferredResult.setErrorResult(errorResponse);
        } else {
            deferredResult.setResult(ResponseEntity.ok(response));
        }
    });
    
    return deferredResult; // 立即返回，不阻塞Tomcat线程
}

// Service层：@Async异步执行
@Async("chatTaskExecutor")
public CompletableFuture<ChatResponse> processChatAsync(ChatRequest request) {
    ChatResponse response = aiCustomerService.processChat(request);
    return CompletableFuture.completedFuture(response);
}
```

#### 线程池配置
```java
@Bean(name = "chatTaskExecutor")
public Executor chatTaskExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(Runtime.getRuntime().availableProcessors() * 2);  // 核心线程
    executor.setMaxPoolSize(Runtime.getRuntime().availableProcessors() * 8);   // 最大线程
    executor.setQueueCapacity(500);                                             // 队列容量
    executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy()); // 拒绝策略
    executor.initialize();
    return executor;
}
```

#### 性能对比
| 模式 | 500并发 | Tomcat线程占用 | 系统吞吐量 |
|------|---------|---------------|-----------|
| 同步 | 大量超时 | 200个全占满 | 低 |
| 异步 | 100%成功 | 接近0 | 高 |

#### Resilience4j 配置

```yaml
resilience4j:
  circuitbreaker:
    instances:
      aiService:
        slidingWindowSize: 20           # 滑动窗口大小
        minimumNumberOfCalls: 10        # 最小调用次数

        slowCallRateThreshold: 80       # 慢调用率阈值80%
        slowCallDurationThreshold: 45s  # 慢调用时间阈值
        waitDurationInOpenState: 30s    # 打开状态等待时间
```

#### 熔断器状态机

```
关闭(Closed) ──失败率>60%──→ 打开(Open)
    ↑                           ↓
    │                    等待30秒后
    │                           ↓
    └────成功率恢复──── 半开(Half-Open)
```

#### 代码实现
```java
@CircuitBreaker(name = "aiService", fallbackMethod = "callAiServiceFallback")

private String callAiService(String fullMessage, ChatRequest request) {

}
```

## 六、面试常见问题与回答

### Q1: 为什么要用二级缓存？

**A:** 
- **L1 Caffeine**：进程内缓存，纳秒级响应，避免网络IO
- **L2 Redis**：分布式缓存，多实例共享，保证数据一致性
- 单级Redis缓存：每次请求都需要网络IO，延迟约10ms
- 二级缓存：热点数据L1直接返回(<1ms)，L2作为兜底，整体性能最优

### Q2: 异步处理解决了什么问题？

**A:**
- **同步问题**：Tomcat默认200个线程，vLLM推理需要20秒，200个线程很快耗尽
- **异步方案**：请求到达后立即释放Tomcat线程，使用独立线程池处理vLLM调用
- **效果**：Tomcat可以持续接收新请求，系统吞吐量从几十提升到260+ TPS

### Q3: 缓存一致性如何保证？

**A:**
- 使用Redis Pub/Sub发布缓存更新消息
- 其他实例订阅消息，收到后清除本地L1缓存
- 下次请求会从L2获取最新数据，并重新填充L1
- 适用于缓存更新频率不高的场景

### Q4: 熔断器的工作原理？

**A:**
- **Closed状态**：正常调用，记录失败率
- **Open状态**：失败率超过60%，直接返回降级响应，不实际调用vLLM
- **Half-Open状态**：等待30秒后进入，允许少量请求探测vLLM是否恢复
- **恢复**：探测成功则回到Closed，失败则继续Open

### Q5: 系统的瓶颈在哪里？如何优化？

**A:**
- **当前瓶颈**：vLLM单实例推理速度慢（20秒/次）
- **优化方向**：
  1. 增加vLLM实例（已实现负载均衡代码，待多GPU环境）
  2. 使用更小的量化模型
  3. 提高缓存命中率（语义缓存、向量数据库）
  4. 增加缓存预热的问题数量
