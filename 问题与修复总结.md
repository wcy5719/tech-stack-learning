## 一、编译与依赖问题

### 1.1 ChatResponse 类缺少 @AllArgsConstructor 注解

**问题描述：**
- 使用 `@Builder` 但缺少 `@AllArgsConstructor`，导致 Lombok 无法生成正确的构造函数
- 编译错误：`ChatResponse.builder()` 无法使用

**修复方案：**
```java
// 修复前
@Data
@NoArgsConstructor
@Builder
public class ChatResponse { ... }

// 修复后
@Data
@NoArgsConstructor
@AllArgsConstructor
@Builder
public class ChatResponse { ... }
```

**影响文件：** `src/main/java/com/wc/aiservice/model/ChatResponse.java`

---

### 1.2 Spring AI 版本兼容性问题

**问题描述：**
- `spring-ai-starter-model-openai` 版本 2.0.0-M4 依赖 `org.springframework.core.retry.RetryTemplate`
- 该模块属于 Spring Framework 7，与当前 Spring Boot 3.5.11（Spring Framework 6.x）不兼容
- 启动报错：`ClassNotFoundException: org.springframework.core.retry.RetryTemplate`

**修复方案：**
- 完全移除 Spring AI 依赖
- 直接使用 WebClient 调用 vLLM API
- 移除相关的自动配置排除

```xml
<!-- 移除的依赖 -->
<dependency>
    <groupId>org.springframework.ai</groupId>
    <artifactId>spring-ai-starter-model-openai</artifactId>
    <version>2.0.0-M4</version>
</dependency>
```

**影响文件：** `pom.xml`

---

### 1.3 maven-compiler-plugin 配置冗余

**问题描述：**
- compiler plugin 使用了嵌套的 `<executions>` 配置，不必要且复杂

**修复方案：**
```xml
<!-- 简化为标准配置 -->
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <configuration>
        <annotationProcessorPaths>
            <path>
                <groupId>org.projectlombok</groupId>
                <artifactId>lombok</artifactId>
                <version>${lombok.version}</version>
            </path>
        </annotationProcessorPaths>
    </configuration>
</plugin>
```

**影响文件：** `pom.xml`

---

### 1.4 缺少 spring-boot-starter-test 依赖

**问题描述：**
- 测试类使用了 JUnit 5 和 Spring Boot Test，但 pom.xml 中没有声明

**修复方案：**
```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-test</artifactId>
    <scope>test</scope>
</dependency>
```

## 三、代码逻辑问题

### 3.1 VllmService WebClient 初始化时机错误

**问题描述：**
- `@Value` 字段注入发生在构造器之后
- WebClient 在构造器中使用 `vllmApiUrl` 时还是 `null`
- 导致连接地址错误：`Connection refused: /[0:0:0:0:0:0:0:1]:80`

**修复方案：**
```java
// 修复前
public VllmService(@Value("${vllm.api.url:http://localhost:8000}") String vllmApiUrl) {
    this.webClient = WebClient.builder().baseUrl(vllmApiUrl).build(); // vllmApiUrl 为 null
}

// 修复后 - 使用 @PostConstruct
@Value("${vllm.api.url:http://localhost:8000}")
private String vllmApiUrl;

private WebClient webClient;

@PostConstruct
public void init() {
    this.webClient = WebClient.builder().baseUrl(vllmApiUrl).build();
}
```

**影响文件：** `src/main/java/com/wc/aiservice/service/VllmService.java`

### 3.2 缓存键前缀重复

**问题描述：**
- `AiCustomerService` 添加前缀 `"ai_response:"`
- `TwoLevelCacheService` 又添加前缀 `"ai_response:"`
- 导致缓存键变成 `"ai_response:ai_response:..."`
- get/put 使用不同的键，缓存无法命中

**修复方案：**
```java
// AiCustomerService.java - 移除前缀
String cacheKey = intent + ":" + filterResult.filteredContent().hashCode();

// TwoLevelCacheService.java - 保持添加前缀
redisTemplate.opsForValue().set(REDIS_KEY_PREFIX + key, json, ...);
```

**影响文件：** 
- `src/main/java/com/wc/aiservice/service/AiCustomerService.java`
- `src/main/java/com/wc/aiservice/service/TwoLevelCacheService.java`

---

### 3.3 布隆过滤器逻辑反转

**问题描述：**
```java
// 错误逻辑
if (!bloomFilterService.mightContain(cacheKey)) {
    // 只有不在布隆过滤器中才查缓存 - 完全反了
    ChatResponse cachedResponse = twoLevelCacheService.get(cacheKey);
    ...
}

// 正确逻辑
if (bloomFilterService.mightContain(cacheKey)) {
    // 在布隆过滤器中才查缓存
    ChatResponse cachedResponse = twoLevelCacheService.get(cacheKey);
    ...
}
```

**修复方案：**
- 去掉 `!` 运算符
- 先检查布隆过滤器，可能存在再查缓存
- 无论是否命中都更新布隆过滤器

**影响文件：** `src/main/java/com/wc/aiservice/service/AiCustomerService.java`

---

### 3.4 幂等性检查顺序错误

**问题描述：**
- 幂等性检查在缓存检查之前执行
- 相同请求被误判为"重复请求"而拦截
- 缓存永远无法命中

**修复方案：**
```java
// 修复后的顺序
public ChatResponse processChat(ChatRequest request) {
    // 1. 敏感词过滤
    // 2. 意图识别
    // 3. 缓存检查（优先）
    if (bloomFilterService.mightContain(cacheKey)) {
        ChatResponse cachedResponse = twoLevelCacheService.get(cacheKey);
        if (cachedResponse != null) {
            cachedResponse.setTimestamp(System.currentTimeMillis());
            return cachedResponse; // 直接返回缓存
        }
    }
    
    // 4. 幂等性检查（仅在实际处理前）
    if (!idempotencyService.checkAndSetIdempotencyKey(idempotencyKey)) {
        // 重复请求也先查缓存
        ChatResponse cachedResponse = twoLevelCacheService.get(cacheKey);
        if (cachedResponse != null) {
            return cachedResponse;
        }
        return "检测到重复请求，请稍后再试";
    }
    
    // 5. 实际 AI 推理
}
```

**影响文件：** `src/main/java/com/wc/aiservice/service/AiCustomerService.java`

### 3.5 ChatResponse cached 字段序列化问题

**问题描述：**
- `private boolean cached = false;` 在 JSON 序列化时默认值处理不当
- Lombok 生成 `isCached()` 但 Jackson 期望 `getCached()`

**修复方案：**
```java
@JsonProperty("cached")
@Builder.Default
private boolean cached = false;
```

**影响文件：** `src/main/java/com/wc/aiservice/model/ChatResponse.java`

---

### 3.6 TwoLevelCacheService Redis 序列化问题

**问题描述：**
- 使用 GenericJackson2JsonRedisSerializer 序列化 ChatResponse 对象
- 反序列化时 instanceof ChatResponse 判断可能失败

**修复方案：**
```java
// 使用 JSON 字符串序列化
public void put(String key, ChatResponse value) {
    localCache.put(key, value);
    String json = objectMapper.writeValueAsString(value);
    redisTemplate.opsForValue().set(REDIS_KEY_PREFIX + key, json, ...);
}

// 反序列化时处理 String 类型
private ChatResponse getFromRedis(String key) {
    Object value = redisTemplate.opsForValue().get(REDIS_KEY_PREFIX + key);
    if (value instanceof String) {
        return objectMapper.readValue((String) value, ChatResponse.class);
    }
    if (value instanceof ChatResponse) {
        return (ChatResponse) value;
    }
    return null;
}
```

**影响文件：** `src/main/java/com/wc/aiservice/service/TwoLevelCacheService.java`

---

### 3.7 缓存返回时未标记 cached 标志

**问题描述：**
- 从缓存获取的响应对象没有设置 `cached=true`
- 前端无法区分是缓存响应还是新推理

**修复方案：**
```java
public ChatResponse get(String key) {
    ChatResponse response = localCache.getIfPresent(key);
    if (response != null) {
        response.setCached(true); // 标记缓存命中
        return response;
    }
    
    response = getFromRedis(key);
    if (response != null) {
        response.setCached(true); // 标记缓存命中
        return response;
    }
    return null;
}
```

## 九、已知限制与优化建议

### 9.1 当前限制
1. vLLM 单实例运行（无负载均衡）
2. Redis Pub/Sub 不支持消息持久化
3. 缓存预热功能未实现
4. WebSocket 不支持流式输出

### 9.2 优化建议
1. **vLLM 多实例**：使用 Nginx 负载均衡
2. **消息队列**：引入 Redisson/RocketMQ 替代 Redis Pub/Sub
3. **缓存预热**：启动时加载常见问题到缓存
4. **流式响应**：实现 SSE 或 WebSocket 流式输出
5. **监控增强**：集成 ELK 日志收集系统
